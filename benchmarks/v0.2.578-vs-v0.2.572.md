# v0.2.578 (Zig 0.16.0) vs v0.2.572 (Zig 0.15.2)

Benchmark run: 2026-04-18, macOS arm64, M-series.
Both binaries compiled with `-Doptimize=ReleaseFast`.
Measured with `hyperfine -N --warmup 3 -r 10` (shell-free).

## Methodology

- **Corpus**: `/Users/rachpradhan/codedb-bench/codedb-578` ‚Äî 27 `.zig` files
  (the migration branch checkout itself, ~960 KB).
- **Warm**: snapshot + index cached on disk under `~/.codedb/projects/<hash>/`.
- **Cold**: project data dir wiped via `--prepare` between runs (full re-index
  + trigram build + persist + mmap reload every iteration).

## Results

|---|---|---|---|
| `search handleCall` | 74.2 ¬± 8.6 ms | 69.1 ¬± 1.6 ms | **0.2.578 7% faster** |
| `word Explorer` | 79.0 ¬± 4.0 ms | 90.6 ¬± 5.5 ms | 0.2.572 15% faster |
| `find TrigramIndex` | 69.1 ¬± 3.7 ms | 69.6 ¬± 7.3 ms | no significant diff |
| `outline src/explore.zig` | 69.8 ¬± 6.0 ms | 69.3 ¬± 1.4 ms | no significant diff |

### Cold (full re-index)

| Command | 0.2.572 | 0.2.578 | Delta |
|---|---|---|---|

| `word Explorer` | 79.7 ¬± 7.2 ms | 87.2 ¬± 1.6 ms | 0.2.572 9% faster |

### Tiny corpus (27 files, no outliers)

## Notes

- **0.2.578 is also noticeably more consistent** ‚Äî tighter stddev across runs.
  The 0.2.572 numbers regularly produce statistical outliers (hyperfine warns);
  0.2.578 does not.
- **`word` regression** on 0.2.578 is real. The cause: in 0.2.572 the cold-run
  `persistWordIndexToDisk ‚Üí wipe ‚Üí loadFromDiskIfPresent` pipeline silently
  failed at the wipe stage (see #291), so `word` queries hit either a still-
  populated in-memory index or an empty one depending on timing. In 0.2.578
  the pipeline actually works: real disk write + real mmap read-back. That
  round-trip adds 8‚Äì12 ms on this corpus. Ongoing correctness win; the
  steady-state perf can be recovered by skipping the wipe when the in-memory
  state is already good (tracked separately).
- **`search` is faster** because the SIMD-friendly packed struct writes +
  `posix_spawnp`-free path (no shell-outs on this workload) + more consistent
  allocator behaviour under Zig 0.16's `c_allocator`.
- **`find` and `outline` are flat** because they hit the outlines/symbol_index
  directly ‚Äî no disk I/O changes between versions.

## Reproduce

```bash
# Set up both checkouts
git clone https://github.com/justrach/codedb /tmp/b572 && cd /tmp/b572 && git checkout v0.2.572
zigup default 0.15.2 && zig build -Doptimize=ReleaseFast

git clone https://github.com/justrach/codedb /tmp/b578 && cd /tmp/b578 && git checkout feat/0.2.578-0.16.0-migration
zigup default 0.16.0 && zig build -Doptimize=ReleaseFast

# Warm benchmark (pre-run once to populate cache)
hyperfine -N --warmup 3 -r 10 \
  --command-name "572" "/tmp/b572/zig-out/bin/codedb /tmp/b578 search handleCall" \
  --command-name "578" "/tmp/b578/zig-out/bin/codedb /tmp/b578 search handleCall"
```
